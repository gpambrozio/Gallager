# Emoji Data Shipping: `emoji.tsv` + SPM `.embedInCode`

**Date:** 2026-07-03
**Context:** PR #632 (issue #630), branch `claude/issue-630`
**Status:** Abandoned — see Outcome below. Kept for the record.

## Problem

PR #632 ships the shared emoji index as `EmojiData.swift`: a ~117KB tab-separated
table inside a Swift string literal, generated by `scripts/generate-emoji-data.py`
and parsed at runtime by `EmojiDatabase`. The string-literal-in-source approach
exists for one hard constraint: the single-file `GallagerCLI` binary is copied
bare into `Gallager.app/Contents/MacOS/` (no resource bundle travels with it),
so `Bundle.module` is unavailable to the CLI.

Concerns raised: runtime parse/memory cost, and repo hygiene (a 2,000-line
generated pseudo-Swift file with lint/format suppressions).

## Measurements (2026-07-03, M-series Mac, `swiftc -O`)

| Axis | Current design |
|---|---|
| Parse | 29ms cold / 12ms warm — once per process, lazy (`EmojiDatabase.shared`) |
| Search | ~1µs warm, 0.3ms first call |
| Heap after parse | ~245KB string payload (+ object overhead) |
| Binary footprint | ~117KB constant data |
| Compile | 0.1s (string literal) vs 4.2s (`.embedInCode`-style `[UInt8]` literal, same payload) |

Conclusion: the runtime axis is a non-problem — the parse is lazy, one-time, and
gesture-triggered (picker open) or paid by a short-lived CLI command. The real
wart is hygiene: generated data masquerading as source. Rejected alternatives:
codegen'd `[Emoji(...)]` literals (type-checker blowup + still runs an
initializer), resource bundle (breaks the single-file CLI or needs a fragile
bundle-copy phase), custom zero-parse binary format (unsafe-pointer complexity
disproportionate to 29ms/245KB), SQLite (overkill), custom SPM build-tool plugin
(new tool to maintain for marginal gain over `.embedInCode`).

## Decision

Commit the data as data and let SPM embed the bytes in code at build time.

1. **Data file:** `ClaudeSpyPackage/Sources/GallagerEmoji/Resources/emoji.tsv`,
   committed. Same rows as today's literal (glyph, label, `|`-joined keywords,
   group, version — tab-separated), plus a `#`-prefixed header comment naming
   the generator. Line-per-emoji keeps regeneration diffs readable.
2. **Package wiring:** the `GallagerEmoji` target gains
   `resources: [.embedInCode("Resources/emoji.tsv")]`. SPM generates
   `PackageResources.emoji_tsv: [UInt8]` (internal to the module) in
   DerivedData; the bytes compile into every binary that links the module —
   macOS app, iOS app, and the bare `GallagerCLI`. `EmojiData.swift` is deleted.
3. **Parser:** `EmojiDatabase.parse` reads
   `String(decoding: PackageResources.emoji_tsv, as: UTF8.self)` and explicitly
   skips `#`-prefixed lines (intentional, documented — not the silent
   field-count fallthrough). `Emoji`, search, categories, and all existing
   tests are untouched.
4. **Generator:** `generate-emoji-data.py` emits plain TSV, deleting its
   Swift-escaping code. The unescaped-`|` emoticon bug (PR #632 review
   comment, e.g. CLDR's `:|`) is fixed in the same rework since it touches the
   same escaping lines.

## Verification

- `GallagerEmojiTests` pass (13 tests, including table-integrity checks).
- Both apps build clean — this also proves Xcode's SPM integration handles
  `.embedInCode` (the one integration risk; supported since Xcode 15).
- The CLI embedded in the built app bundle resolves `find-emoji trash` → 🗑️,
  re-proving the single-file constraint that motivated the original design.
- Accepted cost: ~4s added to clean builds of the `GallagerEmoji` target
  (cached; incremental builds unaffected).

## Outcome (added 2026-07-03, post-implementation)

Implemented through Tasks 1–2 of the plan, then **abandoned during verification**:
the macOS `ClaudeSpyServer` workspace build fails with `cannot find
'PackageResources' in scope`. Root cause (Xcode 26.6, confirmed via build-log
inspection; later verified the dynamicization happens **regardless of
resources**): when a resource-bearing package
target is consumed by **two products in the same build graph** — here the app's library stack *and* the
`GallagerCLI` executable product linked by the app target to force its build
for the copy-into-bundle phase — XCBuild silently ignores `.embedInCode`,
generates the legacy `Bundle.module` accessor, spawns a
`ClaudeSpyPackage_GallagerEmoji.bundle` resource-bundle target, and builds the
library as a dynamic framework (`PackageFrameworks/`). Even with a compile
workaround, that packaging would break the single-file CLI at runtime.
Single-client graphs (iOS app; standalone `swift build`/`swift test`;
package-scheme xcodebuild for macOS) all honor `.embedInCode` — which is why
the mechanism looked viable until the real app build ran. Upstream context:
[swift-package-manager#6969](https://github.com/apple/swift-package-manager/issues/6969).

Resolution: reverted to the original committed-`EmojiData.swift` mechanism
(measured as fine in this spec's own benchmarks), keeping the generator's
`KEYWORD_SEP` escaping fix.
